## Feature 1: Company Brain

### Overview

Automatic knowledge extraction from agent execution transcripts. After a heartbeat run completes, the system analyzes the transcript and extracts structured facts into the knowledge base. Future agents automatically receive relevant knowledge via the existing injection system.

### Extraction Trigger — Hybrid Model

Extraction runs automatically when a `heartbeat_run` completes with `status=success` AND meets at least one threshold:

| Threshold | Default | Configurable |
|---|---|---|
| Run duration | > 120 seconds | Yes, via instance_settings |
| Run cost | > $0.50 | Yes, via instance_settings |
| Agent role | ceo, researcher, lead | Yes, via instance_settings |

Additionally, a manual "Extract Knowledge" button on the run detail page allows extraction for any run regardless of thresholds.

### Extraction Pipeline

```
completeLocalRun(run) — existing function
  ↓
  if (run.status === 'success') → brainService.maybeExtract(run)
    ↓
    evaluateThreshold(run, settings) → boolean
    ↓ (true)
    extractKnowledge(run)
      Input:  run.stdout_excerpt + run.result_json + run.context_snapshot
      Model:  gemini-2.5-flash-lite (same as existing analysis)
      Prompt: Extract 1-10 reusable facts as JSON array
      Output: Array<{title, content, category, tags[]}>
    ↓
    deduplicateAndStore(facts, companyId, agentId)
      For each fact:
        1. Query: SELECT id, ts_rank(search_vector, query) as rank
           FROM knowledge_entries
           WHERE company_id = $companyId
             AND source_type = 'brain_auto'
             AND search_vector @@ to_tsquery('english', $titleAndTags)
           ORDER BY rank DESC LIMIT 1
        2. If rank > 0.3 → UPDATE existing entry (content, tags, updated_at)
        3. If no match  → INSERT new entry
```

### Knowledge Entry Fields (brain_auto)

```json
{
  "source_type": "brain_auto",
  "source_ref": "<heartbeat_run_id>",
  "category": "<auto-classified by LLM>",
  "tags": ["<auto-generated>"],
  "pinned": false,
  "metadata": {
    "extracted_from_run": "<run_id>",
    "agent_name": "<agent_name>",
    "issue_identifier": "<issue_id if available>",
    "extraction_model": "gemini-2.5-flash-lite",
    "confidence": "<high|medium|low>"
  }
}
```

### Auto-Correction (Overwrite)

When a new fact matches an existing `brain_auto` entry (ts_rank > 0.3):
- The existing entry's `content`, `tags`, `category`, and `updated_at` are overwritten
- `source_ref` is updated to the newer run
- `metadata.superseded_refs` accumulates previous run IDs for audit trail

### Extraction Prompt

```
Analyze the following agent execution transcript and extract reusable facts
that would help OTHER agents working on similar tasks in the future.

Agent: ${agentName} (${agentRole})
Task: ${issueTitle} — ${issueDescription}
Transcript (excerpt):
${stdoutExcerpt}

Result:
${JSON.stringify(resultJson)}

Extract between 1 and 10 facts. Each fact should be:
- Self-contained (understandable without the original transcript)
- Actionable (useful for a future agent working on a related task)
- Specific (not generic advice)

Return ONLY a valid JSON array:
[{"title": "...", "content": "...", "category": "...", "tags": ["..."]}]

Categories: technical, process, integration, domain, architecture, debugging
```

### Injection (existing system, no changes needed)

`resolveForInjection(companyId, agentId)` already returns:
1. Pinned entries (company-wide)
2. Agent-scoped entries
3. Contextual entries matching the task

Brain-auto entries flow through path 2 and 3 automatically since they are stored in `knowledge_entries` with the same schema.

| Component | Path | Purpose |
|---|---|---|

## Feature 2: Global Search

### Overview

Unified full-text search across issues, agents, projects, knowledge entries, and heartbeat run events. Replaces the current multi-fetch approach in CommandPalette with a single ranked endpoint.

### Database Migration (0064_global_search.sql)

Add `search_vector` tsvector column + auto-update trigger + GIN index to 4 tables:

#### issues
```sql
ALTER TABLE issues ADD COLUMN search_vector tsvector;

CREATE FUNCTION issues_search_vector_update() RETURNS trigger AS $$
BEGIN
  NEW.search_vector :=
    setweight(to_tsvector('english', COALESCE(NEW.title, '')), 'A') ||
    setweight(to_tsvector('english', COALESCE(NEW.identifier, '')), 'A') ||
    setweight(to_tsvector('english', COALESCE(NEW.description, '')), 'B');
  RETURN NEW;
END $$ LANGUAGE plpgsql;

CREATE TRIGGER issues_search_vector_trigger
  BEFORE INSERT OR UPDATE OF title, identifier, description ON issues
  FOR EACH ROW EXECUTE FUNCTION issues_search_vector_update();

CREATE INDEX issues_search_vector_idx ON issues USING gin(search_vector);

-- Backfill existing rows
UPDATE issues SET search_vector =
  setweight(to_tsvector('english', COALESCE(title, '')), 'A') ||
  setweight(to_tsvector('english', COALESCE(identifier, '')), 'A') ||
  setweight(to_tsvector('english', COALESCE(description, '')), 'B');
```

### Unified Search Endpoint

```
GET /companies/:companyId/search?q=<term>&types=<comma-separated>&limit=20
```

**Query processing:**
```typescript
// Same pattern as existing knowledge search
const terms = q.trim().split(/\s+/).map(t => `${t}:*`).join(' & ');
const tsQuery = `to_tsquery('english', $1)`;
```

**SQL (conceptual):**
```sql
(SELECT 'issue' as type, id, title, identifier as subtitle,
        ts_rank(search_vector, query) as score
 FROM issues, to_tsquery('english', $1) query
 WHERE company_id = $2 AND search_vector @@ query)

UNION ALL

(SELECT 'agent' as type, id, name as title, role as subtitle,
        ts_rank(search_vector, query) as score
 FROM agents, to_tsquery('english', $1) query
 WHERE company_id = $2 AND search_vector @@ query)

UNION ALL

(SELECT 'project' as type, id, name as title, description as subtitle,
        ts_rank(search_vector, query) as score
 FROM projects, to_tsquery('english', $1) query
 WHERE company_id = $2 AND search_vector @@ query)

UNION ALL

(SELECT 'knowledge' as type, id, title, category as subtitle,
        ts_rank(search_vector, query) as score
 FROM knowledge_entries, to_tsquery('english', $1) query
 WHERE company_id = $2 AND search_vector @@ query)

UNION ALL

(SELECT 'run' as type, run_id as id, message as title,
        agent_id::text as subtitle,
        ts_rank(search_vector, query) as score
 FROM heartbeat_run_events, to_tsquery('english', $1) query
 WHERE company_id = $2 AND search_vector @@ query)

ORDER BY score DESC
LIMIT $3
```

**Response shape:**
```typescript
interface SearchResult {
  type: 'issue' | 'agent' | 'project' | 'knowledge' | 'run';
  id: string;
  title: string;
  subtitle: string | null;
  score: number;
  url: string; // computed server-side
}
```

### CommandPalette Changes

Current behavior (3 separate fetches) replaced with:

1. Empty query → show quick actions (create issue, create agent, etc.) — unchanged
2. Non-empty query → debounce 300ms → call `GET /search?q=...`
3. Results grouped by `type` in dropdown sections
4. Each section shows max 5 results, sorted by score
5. Click navigates to the entity's detail page using `url` field

## Dependencies Between Features

Company Brain produces `knowledge_entries` with `source_type='brain_auto'` → Global Search automatically indexes them via the existing `knowledge_entries.search_vector`. The features are independent but synergistic.
