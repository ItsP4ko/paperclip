A — Tab de Sprints en ProjectDetail

Un tab "Sprints" se agrega al header del proyecto. Adentro: sidebar izquierda con backlog + lista de sprints, área principal con el board o vista de planificación. Todo dentro del proyecto.

  • Máxima visibilidad dentro del proyecto
  • Patrón consistente con los otros tabs
  • Tab más entre los existentes

B — Sub-tabs dentro de "Issues"

El tab de Issues pasa a tener sub-navegación: Backlog / Board Sprint / Métricas. El sprint es una vista más del trabajo, no una sección separada.

  • Integrado con el flujo de issues
  • No agrega tabs al header
  • Sprints queda "escondido" bajo Issues
  • Métricas menos prominentes

C — Tabs separados por vista

Cada concepto tiene su propio tab: Backlog, Board (sprint activo), Sprints (historial), Métricas. Máxima claridad conceptual.

  • Cada vista bien definida
  • Métricas prominentes
  • Muchos tabs en el header
  • Fragmenta la navegación

Mi recomendación: Opción A — un tab "Sprints" con sidebar interna. Sigue el patrón del proyecto y mantiene backlog + board + historial accesibles sin fragmentar el header.
