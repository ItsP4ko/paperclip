## Color

All values are CSS variables from `:root` in `index.css`. Do not use raw hex — reference tokens only.

| Role | CSS Token | Light mode value | Usage |
|------|-----------|-----------------|-------|
| Dominant (60%) | `--background` | `oklch(1 0 0)` (white) | Page background, form input backgrounds |
| Secondary (30%) | `--card` | `oklch(1 0 0)` + `--border` edge | Card surface (`bg-card border border-border`) |
| Muted surface | `--muted` | `oklch(0.97 0 0)` | Info boxes inside cards (`bg-muted/30`) |
| Accent (10%) | `--primary` | `oklch(0.205 0 0)` (near-black) | Primary CTA button only |
| Destructive | `--destructive` | `oklch(0.577 0.245 27.325)` (red) | Inline error text only |

Accent (`--primary`) is reserved for: **the primary submit button in the form** (`Button` component default variant). No other elements use `--primary` on these surfaces.

Destructive is reserved for: **inline error paragraphs** (`text-destructive`). No destructive button variant appears in this phase — there are no delete/remove actions.

Muted foreground (`--muted-foreground`) is used for: secondary paragraph text, field labels, placeholder text, expiry lines.

## States and Interactions

This phase touches exactly three interaction states in `InviteLanding.tsx`:

### State 1: Invite loading / health / session loading
- Show: `<div className="mx-auto max-w-xl py-10 text-sm text-muted-foreground">Loading invite...</div>`
- No change in Phase 4 — leave as-is.

### State 2: requiresAuthForHuman = true (authenticated mode, no session)
- Show auth prompt box: `bg-muted/30 rounded-md border border-border p-3 text-sm`
- Button inside: `variant="outline" size="sm"` linking to `/auth?next=/invite/TOKEN`
- Copy (existing, no change): "Sign in or create an account before submitting a human join request."
- Button copy (existing, no change): "Sign in / Create account"
- **Phase 4 does not change this state.** It already works correctly per CONTEXT.md.

### State 3: Submit button — human join
- **CHANGED in Phase 4:** Button label changes from `"Submit join request"` to `"Join [companyName]"` when `joinType === "human"` and `companyName` is truthy.
- Fallback when companyName is null/empty: `"Join"`
- Pending state: `"Submitting…"` (existing, no change)
- Agent join: button copy stays `"Submit join request"` (no change)
- Bootstrap invite: button copy stays `"Accept bootstrap invite"` (no change)

### State 4: onSuccess navigation — approved join
- **NEW in Phase 4:** When `payload.status === 'approved'`, navigate immediately to `/` — skip the `setResult` call entirely.
- No UI state rendered; this is a navigation side-effect only.
- No loading spinner or transition screen — the navigation itself is instant.

### State 5: Join request pending (agent joins, non-auto-approved)
- Show existing join-request-submitted card — no change in Phase 4.

### State 6: Error state
- Show: `<p className="mt-3 text-sm text-destructive">{error}</p>` — existing pattern, no change.

## Copywriting Contract

All copy below applies to the `joinType === "human"` branch in `InviteLanding.tsx` unless noted.

| Element | Copy | Condition | Source |
|---------|------|-----------|--------|
| Primary CTA — human, has company name | `Join [companyName]` | `joinType === "human" && companyName` | CONTEXT.md decision |
| Primary CTA — human, no company name | `Join` | `joinType === "human" && !companyName` | CONTEXT.md decision |
| Primary CTA — pending | `Submitting…` | `acceptMutation.isPending` | Existing, unchanged |
| Primary CTA — agent | `Submit join request` | `joinType === "agent"` | Existing, unchanged |
| Primary CTA — bootstrap | `Accept bootstrap invite` | `invite.inviteType === "bootstrap_ceo"` | Existing, unchanged |
| Page heading — human (has company name) | `Join [companyName]` | `invite.inviteType !== "bootstrap_ceo" && companyName` | Already implemented in InviteLanding h1 |
| Page heading — human (no company name) | `Join this Paperclip company` | `invite.inviteType !== "bootstrap_ceo" && !companyName` | Already implemented |
| Subtext — invite expiry | `Invite expires [datetime].` | always | Existing, unchanged |
| Error state | `[error message from server]` | `error !== null` | Existing pattern; no custom copy needed |
| Auth prompt body | `Sign in or create an account before submitting a human join request.` | `requiresAuthForHuman` | Existing, unchanged |

No empty states are introduced in this phase. The invite-not-found card ("Invite not available") is existing and unchanged.

No destructive actions exist in this phase. No confirmation dialogs are required.

## Accessibility

- All interactive elements already have `touch-action: manipulation` via the global CSS rule.
- `min-height: 44px` on coarse pointer devices is handled globally for buttons and inputs.
- The "Join [Company]" button must remain a `<Button>` (not a link) — it submits a mutation, not a navigation.
- `aria-disabled` pattern: follow Auth.tsx precedent — `disabled={mutation.isPending}` + `aria-disabled={!canSubmit || mutation.isPending}`.
- Focus management: no change required. `autoFocus` is Auth.tsx's concern; InviteLanding does not use `autoFocus`.

## Interaction Summary (Phase 4 Delta Only)

Changes from baseline to implement:

1. **InviteLanding.tsx — button label:** Replace static `"Submit join request"` with dynamic `companyName ? \`Join ${companyName}\` : "Join"` when `joinType === "human"` and not bootstrap. Agent and bootstrap paths stay unchanged.

2. **InviteLanding.tsx — onSuccess:** Before the existing `setResult(...)` call, add a check: if `payload` has `status === 'approved'`, call `navigate('/')` and return early. Await both `queryClient.invalidateQueries` calls before navigating (per Auth.tsx pattern).

3. **No UI changes to Auth.tsx.** The existing `?next=` redirect, sign-in/sign-up form, and two-column layout are already correct.

4. **No new components, no new routes, no new pages.** This phase is behavioral, not structural.
