> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Phase UI Scope Note

Phase 6 is a pure infrastructure and configuration operation. No new UI components or screens are introduced. The only code change is a one-line backend edit (`packages/db/src/client.ts` pool size cap). The UI contract here governs what the **existing deployed application** must correctly render and respond to when the three-tier deployment (Supabase + Railway + Vercel) goes live. Specifically, the auth sign-up and sign-in flows are the acceptance surface for AUTH-05.

## Color

Source: `src/index.css` CSS custom properties — oklch values mapped to approximate functional roles below.

| Role | CSS Variable | Light Mode Value | Usage |
|------|-------------|-----------------|-------|
| Dominant (60%) | `--background` | `oklch(1 0 0)` — pure white | Page background, form surfaces |
| Secondary (30%) | `--secondary` / `--sidebar` | `oklch(0.97 0 0)` — near-white gray | Cards, sidebar, nav, muted surfaces |
| Accent (10%) | `--primary` | `oklch(0.205 0 0)` — near-black | Primary buttons, active states only (see list below) |
| Destructive | `--destructive` | `oklch(0.577 0.245 27.325)` — red | Destructive actions only |
| Muted text | `--muted-foreground` | `oklch(0.556 0 0)` — mid-gray | Placeholder text, secondary labels, disabled states |
| Border | `--border` | `oklch(0.922 0 0)` — light gray | All input/card borders |

Accent (`--primary`) reserved for:
- Primary CTA buttons ("Sign up", "Sign in")
- Active/selected navigation item background
- Focus ring indicators
- Keyboard focus outlines

Accent is NOT used for: informational badges, status indicators, general interactive hover states (those use `--accent` / `--secondary` at 97% lightness).

Dark mode: all tokens above have corresponding dark-mode overrides in `.dark {}` block. The existing implementation handles both modes. No dark-mode-specific contract changes for this phase.

Phase 6 introduces no new screens. The contract below covers the auth flow that must succeed as the live deployment acceptance test (AUTH-05).

| Element | Copy |

## Interaction Contract

### Auth Flow (AUTH-05 acceptance surface)

| Interaction | Expected Behavior |
|-------------|------------------|
| User lands on sign-in page (Vercel URL) | Page renders within 3s; no JS errors in console; no 404 |
| User submits sign-up form | POST to `VITE_API_URL/api/auth/sign-up/email`; on success → redirect to dashboard |
| User submits sign-in form | POST to `VITE_API_URL/api/auth/sign-in/email`; on success → session cookie set; redirect to dashboard |
| User refreshes after sign-in | `GET /api/auth/get-session` returns valid session; user stays authenticated |
| User navigates directly to `/` or any sub-route | Vercel rewrite rule serves `index.html`; React Router handles routing client-side; no 404 |

### Loading States

- Sign-in/sign-up buttons enter disabled + spinner state on submit; re-enable on response (success or error)
- Page-level transitions use existing `skeleton` component for dashboard content while data loads
