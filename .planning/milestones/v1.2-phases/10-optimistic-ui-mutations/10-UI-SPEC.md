## Design Scope for This Phase

Phase 10 is a **behavior-only upgrade** to existing UI surfaces. No new pages or layouts are introduced. The design contract governs:

1. How the UI communicates an **in-flight optimistic state** (status change, assignee change, subtask creation)
2. How the UI communicates a **mutation failure + rollback** (toast error, revert to previous value)
3. How in-flight mutations **suppress WS-driven invalidations** without any visible indicator to the user

All changes occur within `IssueDetail.tsx`, `IssueProperties` panel, and `LiveUpdatesProvider.tsx`. The existing layout, spacing, typography, and color vocabulary remain unchanged.

## Optimistic State Visual Contract

This section defines the precise visual behavior for each optimistic mutation. It is the primary addendum to the standard design contract for this phase.

### OPTM-01 + OPTM-02: Status and Assignee Change

**In-flight state (onMutate fired, server not yet responded):**
- The status value displayed in `StatusBadge` and `StatusIcon` updates immediately to the new value
- No spinner, no opacity reduction, no disabled state on the trigger element
- The property row looks identical to a settled confirmed state — the optimistic write IS the display
- The `isPending` flag on the mutation is NOT used to render any visual loading indicator for status/assignee changes (these are fast enough that a spinner would flash annoyingly)

**Settled state (onSettled fired):**
- Server response replaces cache — display does not change if server confirms the same value
- If server value differs from optimistic value, the cache update causes a re-render to the server-confirmed value (this is intentional and correct)

**Failure state (onError fired, rollback executed):**
- Cache is restored to the snapshot value — UI reverts to previous status/assignee automatically
- Toast fires: `pushToast({ title: "...", body: "...", tone: "error" })`
- Toast positioning and styling are controlled entirely by the existing `ToastViewport` component — no new error UI is introduced

### OPTM-03: Subtask Creation

**In-flight state (onMutate fired, stub appended to list):**
- The subtask stub appears immediately in the subtask list with the entered title
- The "Add subtask" input remains disabled (`disabled` attribute) while `createSubtask.isPending` is true
- A `Loader2` spinner (14px, `animate-spin`) appears inline next to the submit button — this is the EXISTING pattern already in `IssueDetail.tsx` line 1460
- The stub subtask row renders with the `status: "todo"` StatusIcon and the entered title. All other fields (identifier, assignee, priority) render as blank/empty because the stub has null/empty defaults — this is acceptable and intentional
- The stub identifier renders as empty string `""` — the `text-xs font-mono text-muted-foreground` identifier span is omitted or renders as empty text node for the stub row

**Settled state:**
- `onSettled` fires `invalidateQueries` on the list and detail — server response replaces the stub with the real issue object including a populated identifier

**Failure state:**
- List cache restored to pre-stub snapshot
- `setSubtaskError(...)` sets local error state — error displays inline below the subtask input (existing pattern in the component)
- No toast for subtask creation failure — inline error is sufficient and consistent with existing `setSubtaskError` pattern

### OPTM-04: Rollback Visibility

The user MUST see that a mutation failed. Rollback feedback contract:

| Mutation | Failure Indicator | Copy |
|----------|------------------|------|
| Status change (updateStatus) | Error toast | "Status update failed" / "Could not change status. Try again." |
| Assignee change (updateAssignee) | Error toast | "Assignee update failed" / "Could not change assignee. Try again." |
| Subtask creation (createSubtask) | Inline error below input | "Could not create subtask. Try again." |

Toast `tone: "error"` maps to `--destructive` color in `ToastViewport`.

### OPTM-05: WS Guard — No User-Visible Indicator

The `isMutating` guard in `LiveUpdatesProvider` is invisible to the user. When a WS `activity.logged` event is suppressed because a mutation is in-flight:
- No visual indicator appears
- No "suppressed" state is shown
- The guard is purely internal — it prevents a cache overwrite, not a UI element

## Interaction States Summary

| Surface | Idle | In-Flight | Error | Settled |
|---------|------|-----------|-------|---------|
| Status property row | Shows current status | Shows NEW status immediately (optimistic) | Reverts + error toast | Shows server-confirmed status |
| Assignee property row | Shows current assignee | Shows NEW assignee immediately (optimistic) | Reverts + error toast | Shows server-confirmed assignee |
| Subtask list | Shows existing subtasks | Stub appended immediately | Stub removed (reverted) + inline error | Stub replaced by real issue |
| Subtask "Add" button | Enabled | Disabled + Loader2 spinner | Re-enabled + error shown | Re-enabled, input clears |
| WS activity events (issue detail/list) | Processed normally | Suppressed for detail/list keys; activity/comments still process | N/A | Processed normally |
