<objective>
Add server-side bearer token authentication support for user sessions. This enables mobile browsers (iOS Safari, Android Chrome) to authenticate via `Authorization: Bearer <signed_token>` headers instead of cookies, which fail due to Safari ITP third-party cookie blocking. Also fix WebSocket upgrade to accept user session tokens via query param (currently only accepts agent API keys).

Purpose: Unblock mobile authentication by making the server accept bearer tokens for user sessions across both REST API and WebSocket connections.
Output: Modified server auth files + unit tests proving bearer session resolution works in both actorMiddleware and authorizeUpgrade.
</objective>

Currently, when `authHeader` starts with "bearer ", the middleware jumps directly to board API key lookup (line 85), then agent API key lookup (line 104), then agent JWT (line 112). It never tries `resolveSession`. The fix: after extracting the bearer token (line 79), attempt `resolveSession` first when in `authenticated` mode.

Replace the section starting at line 79 (`const token = authHeader.slice(...)`) through the end of the existing bearer handling (before the final `next()` at line 171) with:

```typescript
const token = authHeader.slice("bearer ".length).trim();
if (!token) {
  next();
  return;
}

// NEW: Try BetterAuth session resolution first for bearer tokens in authenticated mode.
// The bearer plugin converts Authorization: Bearer <signed_token> into a session cookie
// internally, so resolveSession will return a valid user session if the token is a
// signed BetterAuth session cookie (from set-auth-token header).
if (opts.deploymentMode === "authenticated" && opts.resolveSession) {
  let session: BetterAuthSessionResult | null = null;
  try {
    session = await opts.resolveSession(req);
  } catch (err) {
    logger.warn(
      { err, method: req.method, url: req.originalUrl },
      "Failed to resolve bearer session from request",
    );
  }
  if (session?.user?.id) {
    const userId = session.user.id;
    const [roleRow, memberships] = await Promise.all([
      db
        .select({ id: instanceUserRoles.id })
        .from(instanceUserRoles)
        .where(and(eq(instanceUserRoles.userId, userId), eq(instanceUserRoles.role, "instance_admin")))
        .then((rows) => rows[0] ?? null),
      db
        .select({ companyId: companyMemberships.companyId })
        .from(companyMemberships)
        .where(
          and(
            eq(companyMemberships.principalType, "user"),
            eq(companyMemberships.principalId, userId),
            eq(companyMemberships.status, "active"),
          ),
        ),
    ]);
    req.actor = {
      type: "board",
      userId,
      companyIds: memberships.map((row) => row.companyId),
      isInstanceAdmin: Boolean(roleRow),
      runId: runIdHeader ?? undefined,
      source: "bearer_session",
    };
    next();
    return;
  }
}

// Fall through: try board API key, then agent API key, then agent JWT (existing logic unchanged)
const boardKey = await boardAuth.findBoardApiKeyByToken(token);
// ... rest of existing code unchanged ...
```

IMPORTANT: The DB lookup logic (instanceUserRoles + companyMemberships) is identical to the no-bearer session resolution path at lines 44-71. The `source` field changes from `"session"` to `"bearer_session"` to distinguish the auth method in logs.

**Modify authorizeUpgrade in server/src/realtime/live-events-ws.ts**

Currently, when a token is present (line 154 onwards), the function does:
1. Hash the token
2. Look up agent API key by hash
3. If `!key || key.companyId !== companyId` => return null

The fix: instead of returning null when agent key lookup fails, attempt user session resolution via synthetic bearer headers. Replace lines 162-164 (the `if (!key || key.companyId !== companyId) { return null; }` block):

```typescript
if (!key || key.companyId !== companyId) {
  // Not a valid agent API key for this company.
  // Try user session resolution via bearer token (signed BetterAuth session cookie).
  if (opts.deploymentMode === "authenticated" && opts.resolveSessionFromHeaders) {
    const syntheticHeaders = new Headers();
    syntheticHeaders.set("authorization", `Bearer ${token}`);
    let session: BetterAuthSessionResult | null = null;
    try {
      session = await opts.resolveSessionFromHeaders(syntheticHeaders);
    } catch (err) {
      logger.warn({ err, companyId }, "Failed to resolve WS bearer session");
      return null;
    }
    const userId = session?.user?.id;
    if (!userId) return null;

    const [roleRow, memberships] = await Promise.all([
      db
        .select({ id: instanceUserRoles.id })
        .from(instanceUserRoles)
        .where(and(eq(instanceUserRoles.userId, userId), eq(instanceUserRoles.role, "instance_admin")))
        .then((rows) => rows[0] ?? null),
      db
        .select({ companyId: companyMemberships.companyId })
        .from(companyMemberships)
        .where(
          and(
            eq(companyMemberships.principalType, "user"),
            eq(companyMemberships.principalId, userId),
            eq(companyMemberships.status, "active"),
          ),
        ),
    ]);

    const hasCompanyMembership = memberships.some((row) => row.companyId === companyId);
    if (!roleRow && !hasCompanyMembership) return null;

    return {
      companyId,
      actorType: "board" as const,
      actorId: userId,
    };
  }
  return null;
}
```

The DB lookup pattern (instanceUserRoles + companyMemberships) is identical to the existing no-token session resolution path at lines 126-152. The key difference is that the token is passed via synthetic `Authorization: Bearer` header so the BetterAuth bearer plugin can verify the HMAC signature.
