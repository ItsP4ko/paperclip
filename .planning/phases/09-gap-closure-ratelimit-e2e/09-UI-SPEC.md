# Phase 9 — UI Design Contract

> Visual and interaction contract for Phase 9: Gap Closure — Rate-Limit Fix & E2E Completion.
> Generated by gsd-ui-researcher.

**Phase scope note:** Phase 9 introduces zero new UI components, pages, or user-facing screens. The work is:
1. A one-line backend code fix (`rate-limit.ts`) and matching test update
2. Manual E2E browser verification on the live deployment (tester operates existing UI)
3. Documentation-only updates (REQUIREMENTS.md checkboxes, SUMMARY frontmatter, VERIFICATION.md)

This contract governs the existing design system that the tester will observe during E2E verification, and the inline feedback states (HTTP 429 error, health check response) that may surface during testing. No new Tailwind classes, components, or styles are authored in this phase.

## Phase 9 Interaction Contract

Because this phase is exclusively backend + verification + docs, the interaction contract is observational rather than prescriptive.

### What the tester does (E2E-04)
1. Navigate to a task on the live Vercel deployment
2. Change task status using the inline status selector
3. Attach a file using the file attachment UI — native OS file picker will open
4. Create a subtask using the subtask creation control
5. Reload the page — verify all changes persist

### What the tester does (E2E-06)
1. Open the live Vercel deployment in two browser windows simultaneously
2. In window A: change a task's status
3. In window B: observe the status change appears without a page refresh
4. Evidence: screenshot of both windows side-by-side showing the update

### Rate-limit fix — no UI surface
The `req.path === "/api/health"` fix is pure backend middleware. No UI component renders or displays differently after the fix. Verification is via the test suite (`cd server && npx vitest run src/__tests__/rate-limit.test.ts`) — no browser interaction required.
