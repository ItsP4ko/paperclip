> Visual and interaction contract for Phase 5: Cross-Origin Code Preparation.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Phase UI Scope Assessment

**This phase introduces zero new UI screens, zero new user-facing flows, and zero new components.**

All changes are in backend source files, API utility files, and a new `vercel.json` config file. The design contract exists to:

1. Carry forward the established design system tokens so the executor does not deviate from them while touching `ui/src/api/client.ts`, `ui/src/api/auth.ts`, and the three WebSocket files.
2. Specify error/empty states that surface if the env-var wiring is wrong — these states already exist in the app and must not be altered by Phase 5 work.
3. Declare the `vercel.json` rewrite rule (structural config, no visual surface) as in-scope.

No new components, no new pages, no new interactive flows.

## Color

Tokens are CSS custom properties from `ui/src/index.css`. All color references must use these tokens — no raw hex values in new code.

| Role | Token | Light value | Dark value | Usage |
|------|-------|-------------|------------|-------|
| Dominant (60%) | `--background` | oklch(1 0 0) — white | oklch(0.145 0 0) — near-black | Page background, main surface |
| Secondary (30%) | `--card` / `--sidebar` | oklch(1 0 0) / oklch(0.985 0 0) | oklch(0.205 0 0) / oklch(0.145 0 0) | Cards, sidebar, nav panels |
| Accent (10%) | `--primary` | oklch(0.205 0 0) — near-black | oklch(0.985 0 0) — near-white | See reserved-for list below |
| Destructive | `--destructive` | oklch(0.577 0.245 27.325) — red | oklch(0.637 0.237 25.331) — red | Destructive actions only |
| Muted text | `--muted-foreground` | oklch(0.556 0 0) | oklch(0.708 0 0) | Secondary text, placeholders, empty states |
| Border | `--border` | oklch(0.922 0 0) | oklch(0.269 0 0) | All dividers, outlines, input borders |

**Accent (`--primary`) reserved for:**
- Active/selected sidebar nav item background
- Primary action buttons (button variant="default")
- Focus ring on interactive elements (`--ring`)
- Sidebar primary item indicator

**Phase 5 color constraint:** No new color tokens. No inline hex/oklch values. All Phase 5 code touches backend files and API utilities only — no new color usage is expected. If a new UI element is introduced (e.g. an env-config warning banner), it must use `--destructive` or the existing amber pattern from `DevRestartBanner.tsx` (`border-amber-300/60 bg-amber-50 text-amber-950`).

## Copywriting Contract

Phase 5 introduces no new user-facing copy. The following entries document existing states that Phase 5 must preserve (executor must not alter them).

| Element | Copy | Location |
|---------|------|----------|
| Network error — API unreachable | Existing toast/error boundary copy; do not alter | App-wide error boundary |
| Dev restart banner headline | "Restart Required" | `DevRestartBanner.tsx` line 39 |
| Dev restart banner body — backend changed | "backend files changed since this server booted" | `DevRestartBanner.tsx` line 26 |
| Dev restart banner body — migrations | "pending migrations need a fresh boot" | `DevRestartBanner.tsx` line 24 |
| Dev restart banner body — both | "backend files changed and migrations are pending" | `DevRestartBanner.tsx` line 21 |

**If Phase 5 introduces a new startup-crash message (BETTER_AUTH_SECRET missing in authenticated mode):**
This crash happens server-side and surfaces only in Railway logs — not in the UI. No new UI copy is needed.

**If a cross-origin failure banner is added (future consideration, not in scope):**
Pattern: amber banner, `AlertTriangle` Lucide icon, headline in `text-[12px] font-semibold uppercase tracking-[0.18em]`, body in `text-sm`. Mirror `DevRestartBanner.tsx` exactly.

## Interaction States

No new interaction states are introduced by Phase 5.

The following states are in the existing app and must continue to work correctly after Phase 5 changes:

| State | Trigger | Visual |
|-------|---------|--------|
| Auth loading | Session cookie not yet validated | Existing loading skeleton |
| Auth error | Cookie rejected (wrong SameSite) | Existing error state + redirect to /auth |
| API offline | `VITE_API_URL` wrong or backend not running | Existing fetch error — unhandled promise or error boundary |
| WebSocket disconnect | WS URL resolves to CDN instead of backend | Existing live-updates fallback (no explicit error shown) |

Phase 5 changes must not break any of these — the executor should verify each after implementation.

## Vercel Configuration (Non-Visual)

`vercel.json` is a structural config file with no visual surface. It goes in `ui/` (the frontend package root). The required SPA rewrite rule:

```json
{
  "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }]
}
```

No build override needed — Vercel auto-detects `pnpm build` from `package.json`. Format choice (vercel.json over vercel.ts) is per Claude's Discretion from CONTEXT.md — use `vercel.json` as it is simpler for a single rewrite rule.

## What the Executor Must NOT Change

Phase 5 is a code-infrastructure phase. To preserve the established design contract, the executor must not:

- Remove or alter the `@media (pointer: coarse)` touch-target rules in `index.css`
- Modify any CSS custom property values in `:root` or `.dark`
- Change typography classes in any `.tsx` component file (edits to `client.ts`, `auth.ts`, WebSocket files are code-only changes)
- Introduce any new color tokens, font declarations, or spacing overrides
- Remove the `credentials: "include"` option from the `request()` function in `client.ts` (required for cross-origin cookie auth)
