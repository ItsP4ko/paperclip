The user must create a Railway service and configure it. Claude cannot authenticate with Railway or create services.

**Steps for the user:**

1. Go to https://railway.com/dashboard and sign in (or create an account)
2. Click **New Project** -> **Deploy from GitHub Repo**
3. Select the paperclip repository
4. Railway will auto-detect the `Dockerfile` at the repo root — let it use the default

**Configure environment variables:**

5. Go to the new service -> **Variables** tab
6. Click **RAW Editor** and paste the following (replace placeholders):

```
DATABASE_URL=postgresql://postgres.PROJECT_REF:[YOUR-PASSWORD]@aws-0-REGION.pooler.supabase.com:5432/postgres
BETTER_AUTH_SECRET=PASTE_OUTPUT_OF_openssl_rand_-hex_32
PAPERCLIP_DEPLOYMENT_MODE=authenticated
PAPERCLIP_DEPLOYMENT_EXPOSURE=public
SERVE_UI=false
PAPERCLIP_ALLOWED_HOSTNAMES=YOUR-APP.vercel.app,healthcheck.railway.app
PAPERCLIP_AUTH_PUBLIC_BASE_URL=https://YOUR-APP.up.railway.app
```

Where:
- `DATABASE_URL`: The Supabase session-mode pooler URL from Plan 02 (port 5432, NOT 6543)
- `BETTER_AUTH_SECRET`: Generate with `openssl rand -hex 32` in terminal. This is a 64-character hex string.
- `PAPERCLIP_ALLOWED_HOSTNAMES`: Replace `YOUR-APP.vercel.app` with the actual Vercel domain you will get in Task 2. You can set a placeholder now and update after Vercel deploy. MUST include `healthcheck.railway.app` or Railway health checks will fail with 403.
- `PAPERCLIP_AUTH_PUBLIC_BASE_URL`: Replace with the actual Railway public URL. You can find this in Railway service -> Settings -> Networking -> Public Networking. Generate a domain if none exists.

**Do NOT set these (they are handled by Dockerfile defaults or Railway injection):**
- `NODE_ENV` (Dockerfile sets `production`)
- `HOST` (Dockerfile sets `0.0.0.0`)
- `PORT` (Railway injects this at runtime)

**Configure health check:**

7. Go to service -> **Settings** -> **Health Checks**
8. Set **Health Check Path**: `/api/health`
9. Set **Health Check Timeout**: `120` seconds (migrations may run on first boot)

**Enable public networking:**

10. Go to service -> **Settings** -> **Networking**
11. Under **Public Networking**, click **Generate Domain** (or use a custom domain)
12. Note the public URL (e.g., `https://your-app.up.railway.app`) — this is needed for `PAPERCLIP_AUTH_PUBLIC_BASE_URL` and for Vercel's `VITE_API_URL`

**Trigger deploy:**

13. Railway should auto-deploy from the connected GitHub repo. If not, trigger a manual deploy.
14. Watch the deployment logs in Railway dashboard. Look for:
    - `Server listening on 0.0.0.0:PORT` (success)
    - No `403 Forbidden` on health check requests (if you see this, check PAPERCLIP_DEPLOYMENT_EXPOSURE=public)
    - No database connection errors (if you see these, verify DATABASE_URL is the session-mode pooler URL)

**Verify health check:**

15. Once Railway shows the deployment as "Active", run:
    ```
    curl -s https://YOUR-APP.up.railway.app/api/health | head -1
    ```
    Expected: HTTP 200 with JSON response containing `"status": "ok"` or similar.

The user must create a Vercel project and configure it. Claude cannot authenticate with Vercel dashboard for initial project creation.

IMPORTANT: `VITE_API_URL` is a BUILD-TIME variable. It must be set in Vercel BEFORE the first deployment, or the build will bake in empty/undefined values and all API calls will fail.

**Steps for the user:**

1. Go to https://vercel.com/dashboard and sign in (or create an account)
2. Click **Add New...** -> **Project**
3. **Import Git Repository**: Select the paperclip repository
4. **Configure Project:**
   - **Root Directory**: Click "Edit" and set to `ui`
   - **Framework Preset**: Should auto-detect as `Vite` (if not, select Vite manually)
   - **Build Command**: `pnpm run build` (default should work)
   - **Output Directory**: `dist` (default for Vite)
   - **Install Command**: `pnpm install --frozen-lockfile`

5. **Environment Variables** (set BEFORE clicking Deploy):
   - Name: `VITE_API_URL`
   - Value: `https://YOUR-APP.up.railway.app` (the Railway public URL from Task 1, NO trailing slash)
   - Environment: Production (and optionally Preview)

6. Click **Deploy**

**If pnpm install fails with workspace resolution errors:**
   - Change Root Directory back to `/` (repo root)
   - Set Build Command to: `pnpm --filter @paperclipai/ui build`
   - Set Output Directory to: `ui/dist`
   - Redeploy

**After deployment completes:**

7. Note the Vercel deployment URL (e.g., `https://your-app.vercel.app`)
8. Go back to Railway and update `PAPERCLIP_ALLOWED_HOSTNAMES` to include the actual Vercel domain:
   ```
   PAPERCLIP_ALLOWED_HOSTNAMES=your-app.vercel.app,healthcheck.railway.app
   ```
   (Replace `your-app.vercel.app` with the actual Vercel domain)
9. Railway will auto-redeploy with the updated variable. Wait for it to become Active again.

**Verify Vercel deployment:**

10. Visit the Vercel URL in a browser
11. The app should load (React SPA renders)
12. Direct navigation to any route (e.g., `https://your-app.vercel.app/some-path`) should NOT return 404 — the `vercel.json` rewrite rule handles this
13. Open browser DevTools -> Network tab. Check that API requests are going to the Railway URL (not relative paths)

All environment variables are wired. CORS, BetterAuth cookies (SameSite=None; Secure), boardMutationGuard, and trustedOrigins are all configured from Phase 5 code + Phase 6 env vars.

The user must verify the auth flow works end-to-end by performing these steps in a browser:

**Sign-up test:**

1. Open the Vercel URL in a browser (use Chrome or Firefox)
2. Navigate to the sign-up page
3. Enter a test email and password (e.g., test@example.com / TestPassword123!)
4. Click "Create account"
5. Expected: Account is created, user is redirected to the dashboard or onboarding flow
6. Check: No errors in the browser console (especially no CORS errors or 403s)
7. Check: In DevTools -> Application -> Cookies, there should be a BetterAuth session cookie with `SameSite=None; Secure` attributes

**Sign-in persistence test:**

8. If sign-up succeeded, sign out first
9. Sign in with the same credentials
10. Expected: Sign-in succeeds, user sees the dashboard
11. **Refresh the page** (F5 or Cmd+R)
12. Expected: User remains signed in (session cookie persists across refresh)
13. Check: `GET /api/auth/get-session` in Network tab returns 200 with session data (not 401 or null)

**Troubleshooting if sign-up/sign-in fails with 403:**
- Check Railway logs for "untrusted origin" errors
- Verify `PAPERCLIP_ALLOWED_HOSTNAMES` includes the exact Vercel domain (no https://, just the hostname)
- Verify `PAPERCLIP_AUTH_PUBLIC_BASE_URL` is set to the Railway public URL (with https://)

**Troubleshooting if cookies are not set (sign-in returns 200 but session is null on refresh):**
- This may be the `vercel.app` public suffix issue (see Research pitfall #3)
- Check browser DevTools -> Application -> Cookies for any blocked cookie warnings
- If this happens, a custom domain will be needed (defer to Phase 7)

Resume signal: Type "auth works" if sign-up, sign-in, and session persistence all succeed. Type "auth fails" with the specific error if something does not work (include browser console errors and Railway logs).
